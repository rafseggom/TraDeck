\section{Arquitectura del Backend: Smart Contracts}
El núcleo de TraDeck se fundamenta en una arquitectura de contratos inteligentes componibles, desarrollados en Solidity y desplegados sobre la red Ethereum (simulada en entorno local mediante Hardhat). El ecosistema ha evolucionado hacia una economía basada en un token de utilidad propio, separando la lógica financiera de la lógica de propiedad.

\subsection{TraDeckCoin (TDKC) - Capa Financiera}
Para mitigar la volatilidad inherente a las criptomonedas nativas (como el Ether), se ha implementado un contrato que hereda el estándar ERC-20. Este contrato actúa como la entidad emisora del token de utilidad TDKC, que funciona como la divisa exclusiva del mercado. \\

Se ha integrado una función pública de distribución inicial (\textit{Airdrop}), que actúa como un mecanismo de adopción de usuarios, inyectando liquidez inicial (1.000 TDKC) a las nuevas carteras que interactúan con la plataforma.

\subsection{TraDeckNFT - Mercado y Custodia (Escrow)}
El contrato principal hereda el estándar ERC-721 y gestiona tanto la emisión (\textit{Minting}) de los activos como su intercambio seguro. Para garantizar la confianza en un entorno P2P sin intermediarios humanos, se ha implementado un patrón de diseño de custodia o \textit{Escrow}. \\

La máquina de estados del mercado sigue el siguiente flujo transaccional:
\begin{enumerate}
    \item \textbf{Listado y Bloqueo:} Al invocar \texttt{listForSale}, el vendedor transfiere temporalmente la propiedad de su NFT al propio contrato inteligente (\texttt{address(this)}). Esto garantiza al comprador que el activo existe y está disponible.
    \item \textbf{Compromiso de Compra:} El comprador ejecuta \texttt{buyCard}, depositando los fondos necesarios. El contrato retiene tanto el NFT como el dinero, alcanzando el estado de \textit{InEscrow}.
    \item \textbf{Liquidación Atómica:} Mediante la función \texttt{confirmDelivery}, se cierra la operación. El contrato ejecuta una transacción atómica donde transfiere los fondos al vendedor y el activo al comprador de forma simultánea. Si cualquiera de las dos transferencias falla, la operación completa se revierte.
\end{enumerate}

\subsection{Seguridad y Mitigación de Riesgos}
Dada la inmutabilidad del código (\textit{bytecode}) una vez desplegado en la blockchain, el contrato \texttt{TraDeckNFT} implementa reglas estrictas de ciberseguridad:
\begin{itemize}
    \item \textbf{Validación de Identidad:} El sistema prescinde de contraseñas tradicionales, utilizando firmas criptográficas (\texttt{msg.sender}) para autorizar cada escritura en la base de datos distribuida.
    \item \textbf{Patrón Checks-Effects-Interactions:} Para prevenir ataques de reentrada (\textit{Reentrancy Attacks}), las funciones que manejan transferencias de valor actualizan o borran el estado interno (ej. \texttt{delete trades[tokenId]}) antes de interactuar con contratos externos o enviar fondos.
\end{itemize}

\section{Aseguramiento de Calidad y Entorno de Ejecución}
La inmutabilidad intrínseca de los contratos inteligentes en Ethereum hace que sea imposible aplicar parches de seguridad convencionales una vez el código ha sido desplegado. Por ello, el ecosistema TraDeck se ha desarrollado bajo una estricta metodología de pruebas (\textit{Testing}).

\subsection{Entorno de Pruebas Unitarias}
Para la simulación y validación de la lógica de negocio, se ha utilizado el \textit{framework} Hardhat en combinación con las librerías Ethers.js y Chai. Esto permitió levantar una red blockchain efímera en memoria y ejecutar pruebas de caja blanca sobre los contratos.

El conjunto de pruebas validó de forma matemática:
\begin{itemize}
    \item \textbf{Control de Acceso:} Verificación de que solo el propietario de un token ERC-721 puede invocar la función \texttt{listForSale}.
    \item \textbf{Aislamiento de Fondos:} Comprobación de que la función \texttt{buyCard} retiene exactamente la cantidad de Wei (la fracción más pequeña de ETH) especificada por el vendedor.
    \item \textbf{Reversión de Estado:} Uso de aserciones para confirmar que transacciones maliciosas (ej. intentar confirmar una entrega sin ser el comprador) disparan los mecanismos \texttt{require} y revierten el estado de la red sin consumir gas innecesario.
\end{itemize}

\subsection{Orquestación del Despliegue con Hardhat Ignition}
El paso a producción (o a redes de prueba) de una arquitectura multi-contrato requiere precisión para mantener las referencias entre ellos. En lugar de utilizar \textit{scripts} de despliegue imperativos, se optó por \textbf{Hardhat Ignition}.

Ignition proporciona un enfoque declarativo, permitiendo definir módulos de despliegue. El motor se encarga automáticamente de:
\begin{enumerate}
    \item Calcular los costes de Gas óptimos para la transacción de creación del contrato.
    \item Firmar las transacciones con las claves privadas configuradas de forma segura.
    \item Resolver dependencias (ej. asegurar que \texttt{TraDeckCoin} se despliega antes de que \texttt{TraDeckNFT} intente interactuar con él).
    \item Generar los artefactos (ABIs) necesarios para la integración con la interfaz de usuario.
\end{enumerate}

\subsection{Simulación Manual (Remix IDE y Nodo Local)}
Como capa final de QA antes de la integración con el \textit{Frontend}, se levantó un nodo local (\texttt{localhost:8545}) persistente. Mediante la conexión de este nodo al entorno de desarrollo Remix IDE vía \textit{Hardhat Provider}, se realizó una auditoría visual. 

Este enfoque empírico permitió actuar bajo múltiples identidades criptográficas (Cuentas 0 a 19 autogeneradas por Hardhat) para testear el flujo P2P en tiempo real, confirmando que la experiencia de intercambio atómico se comporta según las especificaciones del diseño.
